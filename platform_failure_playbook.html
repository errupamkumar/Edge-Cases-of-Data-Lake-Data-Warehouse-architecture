Production Failure Edge Cases

Every failure scenario your platform can encounter, with root cause, business impact, detection signal, and step-by-step recovery. Click any scenario to expand.

⬡
Debezium / CDC — Ingestion Layer
PostgreSQL WAL → Debezium → Kafka Connect → Kafka topic
CRITICAL
WAL replication slot bloat — PostgreSQL disk full
Debezium lag causes PG to retain WAL indefinitely → disk exhaustion → DB crash
▼
What happens
Debezium reads from a PostgreSQL replication slot. If Debezium is down, slow, or restarted with a large gap, PG cannot clean up WAL segments — it must keep them until the slot catches up. At 5K inserts/sec on a busy trips table, WAL can grow at 2–5 GB/hour. Undetected, this fills your data volume and crashes PostgreSQL, taking down your entire transactional system — not just the data pipeline.
Business impact
Service affected
Full DB crash
Max lag before risk
~4 hours
Data loss risk
HIGH if PG crashes
Detection
Monitor pg_replication_slots.confirmed_flush_lsn lag. Alert when WAL retention > 20 GB.
-- Run every 5 min, alert if result > 20000 MB SELECT slot_name, pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)) AS lag_size FROM pg_replication_slots WHERE slot_type = 'logical';
Recovery steps
  1. 1
    Set max_slot_wal_keep_size = 10GB in postgresql.conf — PG will auto-drop the slot before disk is full (Debezium will need full snapshot on reconnect)
  2. 2
    If slot is already bloated, manually drop it: SELECT pg_drop_replication_slot('debezium_slot');
  3. 3
    Restart Debezium — it detects missing slot and performs initial snapshot of current table state
  4. 4
    After snapshot, normal incremental CDC resumes. Gap period is covered by snapshot data.
  5. 5
    Run nightly reconciliation job to find any records missing from Silver during the gap and patch from PostgreSQL full export
Prevention
max_slot_wal_keep_size=10GB Alert: WAL lag > 5GB Debezium health check /q/health PG disk alert at 70%
HIGH
Debezium connector restart causes duplicate CDC events
Connector offset not committed before crash → replays already-processed events
▼
What happens
Debezium commits its WAL offset to Kafka Connect's internal topic (connect-offsets) periodically, not after every event. If the connector crashes between commits, on restart it replays from the last committed offset — sending duplicate events to your Kafka topic. These duplicates flow into Flink, potentially causing double-counting in trip metrics, double payment records in Silver, or incorrect driver earnings.
Business impact
Risk
Double payments
Detection window
Minutes to hours
Self-healing?
No — manual fix
Recovery steps
  1. 1
    Flink deduplication (bloom filter keyed on op + lsn + table) catches most duplicates in the stream path
  2. 2
    Silver layer: use Parquet MERGE/UPSERT on primary key — duplicate CDC events for same PK are idempotent (last write wins)
  3. 3
    After incident, run reconciliation: count events per trip_id in Silver vs PostgreSQL. Flag any trip_id with count > 1
Prevention
offset.flush.interval.ms=1000 Flink: bloom filter dedup on LSN Silver: upsert on PK
HIGH
PostgreSQL schema change breaks Debezium
ALTER TABLE on a captured table without Debezium config update → connector error loop
▼
What happens
A developer runs ALTER TABLE trips ADD COLUMN surge_zone_id VARCHAR(50) on production without notifying the data team. Debezium's internal schema history (debezium-history Kafka topic) does not have this column. The next WAL event that includes this column causes a deserialization failure. Debezium connector enters error state. CDC stops. All new trip data stops flowing to Kafka.
Recovery steps
  1. 1
    Identify the failing connector: GET /connectors/trips-cdc/status — look for FAILED state with schema error
  2. 2
    Update Debezium connector config to include new column in column.include.list
  3. 3
    Restart connector: POST /connectors/trips-cdc/restart
  4. 4
    Debezium will replay from last committed offset — new column will be present in replayed events
  5. 5
    Update Avro schema in Schema Registry for the CDC topic to include the new field (nullable with default null)
Prevention — mandatory DDL process
All schema changes to CDC-captured tables must go through a PR that includes: (1) Debezium config update, (2) Avro schema update with backward-compatible field, (3) Flink operator update to handle nullable field. Gate deployments on this checklist.

Kafka Broker — Message Bus
Topic partitions, replication, consumer groups, offset management
CRITICAL
Kafka broker leader election storm — all partitions unavailable
Simultaneous broker restart / network partition → ISR shrinks → producer timeout
▼
What happens
With replication factor 3 and min.insync.replicas=2, if 2 of 3 brokers go down simultaneously (e.g., a patch restart cascade or rack-level power failure), the GPS topic rides.gps.raw loses quorum. Producers get NOT_ENOUGH_REPLICAS errors. GPS pings are lost for the outage duration — there is no replay possible since the data never entered Kafka.
Business impact
GPS data gap
Permanent loss
Trip tracking
Blind for duration
Surge pricing
Stale data
Recovery steps
  1. 1
    Bring brokers back up — leader election runs automatically, partitions re-assign within seconds
  2. 2
    GPS producers have local device buffers (typically 30–60s). On reconnect, devices flush buffered pings to Kafka — partial recovery
  3. 3
    For CDC data: Debezium was buffered at WAL level — no loss, replays on reconnect
  4. 4
    Mark affected Bronze partitions with a _gap_metadata.json file documenting the time range of missing GPS data
  5. 5
    Downstream (ClickHouse dashboards) — display a data quality banner for affected time windows
Prevention
Brokers across 3 racks/AZs Rolling restarts only (1 broker at a time) GPS device SDK: local 60s buffer + retry Alert: ISR < RF
HIGH
Consumer group rebalance storm — Flink jobs pause for minutes
Flink TaskManager OOM → task restart → Kafka rebalance → all consumers pause
▼
What happens
A Flink TaskManager OOM kills one task. Flink restarts it. The Kafka consumer group detects a member leaving and triggers a rebalance — during which ALL consumers in the group stop processing and wait for partition reassignment. At 500K GPS events/sec, a 30-second rebalance pause accumulates 15M buffered events in Kafka. When consumers resume, they process at 2× speed to catch up, causing a CPU spike that can cascade into more OOMs.
Recovery steps
  1. 1
    Use session.timeout.ms=45000 and max.poll.interval.ms=300000 to reduce false rebalances
  2. 2
    Enable Kafka's incremental cooperative rebalancing: partition.assignment.strategy=CooperativeStickyAssignor — only affected partitions pause, not all
  3. 3
    For the OOM root cause: increase TaskManager heap or reduce Flink state size by compacting RocksDB state
MEDIUM
Topic partition hot-spotting — one partition overwhelmed
All trips from city event keyed to same driver_id prefix → single partition at 10× load
▼
What happens
GPS topic is keyed on driver_id. If a new city launches with 10,000 drivers whose IDs all hash to the same partition bucket (e.g., sequential numeric IDs starting at a round number), one Kafka partition receives all GPS pings for that city. The Flink subtask consuming that partition becomes the bottleneck, building lag while other subtasks sit idle.
Recovery steps
  1. 1
    Identify hot partition: kafka-consumer-groups.sh --describe — look for one partition with 10× the lag of others
  2. 2
    Short-term: increase parallelism for that Flink subtask by adjusting setParallelism() on the consuming operator
  3. 3
    Medium-term: change GPS topic key to murmur2_hash(driver_id + random_salt) to scatter load — note: per-driver ordering is lost, acceptable for GPS pings
  4. 4
    Long-term: use 128 or 256 GPS partitions (not 64) with random sub-key for better distribution

GPS Ingestion API
Driver devices → REST/WebSocket/gRPC → Kafka producer
CRITICAL
GPS ingest service crash — complete GPS data loss for active trips
No fallback buffer → active trip tracking blind until restart
▼
What happens
Unlike Debezium (which has WAL as a replay buffer), GPS pings from devices are fire-and-forget UDP/HTTP. If the ingestion service crashes with no local buffer, pings sent during the outage are lost permanently. Active trips have no location updates — driver matching, ETA, and surge pricing all go blind. At 500K events/sec, even a 10-second outage means 5M lost GPS points.
Recovery steps
  1. 1
    Kubernetes readiness/liveness probes restart the pod within 15–30s automatically
  2. 2
    GPS device SDK should implement exponential backoff retry (10s, 30s, 60s) and local ring buffer for last 60 seconds of pings
  3. 3
    On recovery, device flushes buffered pings with original event_time — Flink event-time semantics handles the out-of-order delivery correctly via watermarks
  4. 4
    Mark the outage time window in Bronze metadata for downstream audit
Prevention
Min 3 replicas for ingestion pods Device SDK: 60s local ring buffer Load balancer health checks Circuit breaker to Kafka
HIGH
GPS timestamp skew — device clocks out of sync
Driver device NTP drift → event_time 2+ minutes ahead/behind → Flink window misassignment
▼
What happens
Flink uses event_time for windowed aggregations. If a driver's phone clock drifts by +5 minutes (common in low-end Android devices without reliable NTP), their GPS pings arrive at Flink with a future timestamp. Flink's watermark (2s late tolerance) will not admit these events into the correct window — they are either dropped to the side output or incorrectly placed in future windows. This causes ETA calculations and zone demand aggregations to be wrong for that driver.
Recovery
  1. 1
    Ingestion API stamps each event with server_ingest_time alongside the device's event_time
  2. 2
    Flink validation operator: if abs(event_time - server_ingest_time) > 120s, replace event_time with server_ingest_time and set clock_corrected=true flag
  3. 3
    KPI: track clock_corrected rate per device — high rate flags devices needing NTP enforcement

Flink Streaming — Processing Layer
Stateful stream processing, checkpointing, exactly-once semantics
CRITICAL
Flink checkpoint failure — state not persisted, recovery replays hours of data
MinIO write timeout during checkpoint → checkpoint fails → on crash, Flink replays from last successful checkpoint
▼
What happens
Flink checkpoints every 30 seconds to MinIO. If MinIO is slow (high load) or the checkpoint takes longer than the timeout (default 10 minutes), the checkpoint fails. If Flink then crashes, it must recover from the last successful checkpoint — which could be hours ago if checkpoints have been failing silently. This means re-processing hours of Kafka events, which causes duplicate outputs to downstream sinks (ClickHouse, MinIO Silver) unless sinks are idempotent.
Business impact
Replay duration
Up to hours
Duplicate risk
High
Self-healing?
Yes, with idempotent sinks
Recovery steps
  1. 1
    Flink automatically restores from last successful checkpoint — no manual action needed for state recovery
  2. 2
    ClickHouse sink: use ReplacingMergeTree engine — replayed events with same primary key are deduplicated at merge time
  3. 3
    MinIO Silver writes: use partition-keyed Parquet with upsert logic — replay overwrites existing Parquet files for the same time partition
  4. 4
    Investigate checkpoint failure root cause: flink.jobmanager.checkpoint.history → look for "checkpoint expired" or "MinIO connection timeout"
Prevention
checkpoint.timeout=600000 (10min) Alert: consecutive checkpoint failures > 3 Incremental checkpoints (RocksDB) MinIO dedicated for checkpoints
CRITICAL
Flink backpressure cascade — entire pipeline stalls
Slow sink (ClickHouse overloaded) → Flink buffers fill → backpressure propagates to Kafka consumers → lag explodes
▼
What happens
Backpressure in Flink propagates upstream. If ClickHouse is slow to accept writes (e.g., a heavy compaction job runs), the Flink→ClickHouse sink blocks. The blocking operator slows upstream operators, which eventually slow the Kafka source consumer. Kafka lag grows. GPS pings queue in Kafka. Surge pricing goes stale. The entire pipeline degrades from one downstream slowness — this is the "slow consumer" problem at pipeline scale.
Detection
Flink Web UI → Job graph: backpressure percentage > 50% shown in red on the affected operator. Also visible as growing Kafka consumer lag for the GPS consumer group.
Recovery steps
  1. 1
    Identify the backpressured operator in Flink Web UI
  2. 2
    For ClickHouse sink backpressure: add an async sink buffer (AsyncSinkBase) that decouples Flink throughput from ClickHouse write speed
  3. 3
    For ClickHouse overload: temporarily disable heavy background merges: SYSTEM STOP MERGES
  4. 4
    Long-term: separate real-time ClickHouse cluster from batch analytics cluster to prevent interference
HIGH
RocksDB state store corruption — keyed state lost after crash
Disk failure on TaskManager node corrupts RocksDB SST files → state unrecoverable
▼
What happens
Flink stores keyed state (per-driver last known position, bloom filters, trip state) in RocksDB on the TaskManager's local disk. If the local disk fails or the node is terminated (cloud preemption), the in-memory/on-disk state is lost. Flink recovers from the MinIO checkpoint — but if the last checkpoint was 30s ago, 30s of state updates (position changes, new trips started) are lost. Driver utilisation counters reset. Active trip state is gone — those trips appear "orphaned."
Recovery steps
  1. 1
    Restore from last MinIO checkpoint (automatic on Flink restart)
  2. 2
    For orphaned trips (trip started in the 30s gap): reconciliation job queries PostgreSQL for trips in status=active with no recent GPS in Silver → re-inject synthetic "trip_resumed" event
  3. 3
    Reduce checkpoint interval to 10s to minimize state loss window for critical state (trip lifecycle)
Prevention
Incremental RocksDB checkpoints to MinIO Checkpoint interval: 10s for trip state ops TaskManagers on SSDs, not HDDs

Flink Batch — Nightly Jobs
Bronze → Silver → Gold aggregation, backfill, reconciliation
HIGH
Nightly Gold job fails at 03:00 — driver payout calculations missing
OOM on large partition → job fails → Gold not updated → payouts run on stale data
▼
What happens
The nightly Flink batch job aggregates Bronze Parquet into Gold summaries (driver earnings, zone heatmaps). On a high-volume day (e.g., New Year's Eve), one partition has 3× normal size. The Flink task processing it runs out of heap. The job fails at 03:00. Gold is not updated. At 06:00, the driver payout service queries Gold for earnings data — it gets yesterday's values. Drivers are paid wrong amounts.
Business impact
Driver payouts
Incorrect
Discovery time
At 06:00 batch run
Recovery time
1–3 hours
Recovery steps
  1. 1
    Increase Flink TaskManager heap for batch jobs: set taskmanager.memory.process.size=8g and resubmit
  2. 2
    Re-run the failed Gold job immediately. Bronze is immutable — it's always available to re-derive from
  3. 3
    Delay driver payout service start time until Gold is confirmed fresh: check gold/_SUCCESS sentinel file in MinIO before proceeding
  4. 4
    Implement partition-level restart: failed jobs checkpoint their progress per-partition, restart from the failed partition only
Prevention
Gold freshness gate before payouts Batch job alert if not completed by 04:30 Adaptive query execution (auto-scale parallelism) Skew detection on partition size pre-job
MEDIUM
Backfill job overwrites current Silver partition — data corruption
Re-run of old backfill job without partition check → overwrites today's Silver data with old logic
▼
What happens
An engineer re-runs a backfill job to fix a 3-day-old data issue but accidentally sets the date range to include today. The job writes to silver/trips_enriched/year=2026/month=04/day=22/ — overwriting today's streaming-produced Silver with batch-produced (and potentially different logic version) data. Today's Gold aggregates become inconsistent with what ClickHouse has already ingested.
Recovery steps
  1. 1
    MinIO versioning enabled: restore overwritten partition from previous version
  2. 2
    Re-run Gold batch job for the affected day with the restored Silver
  3. 3
    Invalidate ClickHouse partition: ALTER TABLE trips DROP PARTITION '202604' and reload from corrected Gold
Prevention
MinIO versioning on Silver/Gold buckets Backfill jobs require explicit --dry-run first Backfill blocked on partitions < 2 days old

Rule Engine — Dynamic Business Logic
Flink broadcast state, PostgreSQL rule store, surge/fraud logic
CRITICAL
Buggy rule deployed → surge multiplier set to 0 for all zones
Rule syntax error passes validation → broadcast to all Flink operators → zero-fare trips issued
▼
What happens
A product manager updates the surge rule in the PostgreSQL rule store. The rule has a logic error: surge_multiplier = demand / available_drivers * 0 (accidentally multiplied by zero). Flink's broadcast stream picks up the new rule within 30s and broadcasts it to all 64 parallel operators. Every trip from that point forward gets surge_multiplier = 0. Revenue per trip drops to base fare × 0 = effectively zero. If undetected for 1 hour at 10,000 trips/hour, the revenue loss is enormous.
Business impact
Revenue loss
Severe / hour
Detection time
Revenue KPI spike
Recovery
Rollback rule + refund
Recovery steps
  1. 1
    Immediately: set rule effective_until = NOW() in PostgreSQL to deactivate the buggy rule
  2. 2
    Restore previous rule version from rule audit table (every rule change is versioned, never deleted)
  3. 3
    Flink picks up the restored rule within 30s — surge pricing corrects forward
  4. 4
    For affected trips: query Silver for trips with surge_mult = 0 during the window → issue fare corrections in PostgreSQL payments table
  5. 5
    Re-run Gold aggregates for affected partitions to correct revenue figures
Prevention — mandatory rule validation pipeline
-- Validation checks before any rule goes live: CHECK surge_multiplier BETWEEN 0.8 AND 10.0 -- range guard CHECK rule_expression NOT CONTAINS '* 0' -- zero-mult guard REQUIRE shadow_mode_test_on_last_24h_data -- dry run first REQUIRE approver != author -- 4-eyes principle
Shadow mode testing before live Range guards on all numeric outputs Revenue KPI alert on >20% drop in 5min 4-eyes approval for rule changes

MinIO / S3 Storage
Bronze, Silver, Gold Parquet/Avro, Flink checkpoints
CRITICAL
MinIO disk full — Bronze writes fail, Flink checkpoints fail simultaneously
Unmonitored disk growth + large GPS volume → disk at 100% → Bronze writes blocked + checkpoints fail → pipeline unstable
▼
What happens
Bronze GPS Avro files write continuously at ~10 TB/day. Without lifecycle policies, Bronze fills the MinIO cluster. When disk hits 100%, two things fail simultaneously: (1) Flink Bronze sink starts returning 503 errors, causing GPS data loss. (2) Flink checkpoint writes to MinIO also fail — Flink cannot persist state. If Flink then crashes, it cannot recover. This is a compounding failure: data loss + unrecoverable state.
Recovery steps
  1. 1
    Emergency: delete oldest Bronze GPS partitions beyond 90-day retention (Bronze GPS is lowest-value storage)
  2. 2
    Flink will auto-recover checkpoints once MinIO has space — restart Flink if needed
  3. 3
    Implement MinIO lifecycle rule immediately: bronze/gps/ → expire after 90 days
Prevention
Alert: MinIO disk > 70% Lifecycle: Bronze GPS expire 90d Lifecycle: Bronze CDC expire 90d Separate MinIO cluster for checkpoints

Schema Evolution Edge Cases
Avro Schema Registry, Parquet schema drift, backward/forward compatibility
HIGH
Breaking schema change published to Schema Registry without compatibility check
Renamed field in GPS Avro schema → all existing consumers fail to deserialize new events
▼
What happens
A developer renames speed_kmh to speed_ms in the GPS Avro schema and registers it with Schema Registry. Flink operators compiled against the old schema cannot deserialize the new events — field not found. The Flink GPS validation operator starts throwing deserialization exceptions. Events route to DLQ. GPS processing stops.
Recovery steps
  1. 1
    Revert schema: register previous schema version as the current one in Schema Registry
  2. 2
    Events already on DLQ: re-process from DLQ topic using a Flink job that handles both schema versions
  3. 3
    Correct approach for rename: add new field speed_ms with alias ["speed_kmh"] — Avro aliases allow both consumers to read it
Prevention
Schema Registry: BACKWARD_TRANSITIVE compatibility Schema changes blocked unless compatible Never rename fields — use aliases Schema change PR requires Flink consumer test

ClickHouse OLAP — Analytics Layer
MergeTree tables, Materialized Views, real-time ingestion, query performance
HIGH
ClickHouse too many parts error — ingestion blocked
High-frequency inserts create too many small parts → CH throws "Too many parts" → writes rejected
▼
What happens
ClickHouse MergeTree writes data in "parts" that are periodically merged in the background. If data arrives faster than ClickHouse can merge (e.g., Flink inserts 1,000 rows/second in tiny batches of 10 rows), the number of active parts explodes. ClickHouse enforces a limit (~300 parts per partition) and throws Too many parts (N). Merges are processing significantly slower than inserts. All writes are rejected. Real-time GPS-to-dashboard pipeline breaks.
Recovery steps
  1. 1
    Emergency: SYSTEM START MERGES trips — ensure merges are running
  2. 2
    Increase batch size in Flink ClickHouse sink: from rows-per-insert to buffer of 10,000 rows or 1s — dramatically reduces part creation rate
  3. 3
    Set max_insert_block_size = 100000 and min_insert_block_size_rows = 100000 in ClickHouse
  4. 4
    Monitor: SELECT count() FROM system.parts WHERE table='trips' AND active — should stay below 100
Prevention
Flink CH sink: batch=10000 rows or 1s Alert: active parts > 200 Buffer table pattern (Kafka engine → MV → final table)
HIGH
Materialized View query returns stale data — surge pricing uses wrong demand figures
MV definition uses wrong ORDER BY → background deduplication gives incorrect sums
▼
What happens
The zone_demand_hourly Materialized View uses SummingMergeTree. This engine merges rows with the same ORDER BY key by summing numeric columns. If the ORDER BY key is wrong (e.g., missing hour), all hours for a zone merge into one row with an enormous incorrect trip count. Surge pricing API queries this MV and gets wildly inflated demand figures — surge fires incorrectly in zones with normal demand, driving away riders.
Recovery steps
  1. 1
    Drop and recreate the MV with correct definition
  2. 2
    Backfill MV from source trips table: INSERT INTO zone_demand_hourly SELECT ... FROM trips WHERE trip_start > now() - INTERVAL 7 DAY
  3. 3
    Validate MV output: cross-check with direct query on trips table for same time window
  4. 4
    Surge pricing API: add sanity check — if MV demand > 3× rolling average for zone, use direct trips query instead

REST API Layer
FastAPI/Go → ClickHouse → Redis cache → downstream consumers
MEDIUM
Redis cache stampede — ClickHouse overwhelmed on cache expiry
All zone demand cache keys expire simultaneously → 1000 concurrent requests hit ClickHouse
▼
What happens
All zone demand cache keys are set with TTL = 30s at the same time (on service startup or after a Redis flush). When they all expire simultaneously, a burst of 1000+ concurrent API requests bypass the cache and query ClickHouse directly. ClickHouse, designed for analytical workloads, is not optimised for 1000 concurrent point queries — it queues them, response times spike to 30s, and the API times out. Surge pricing UI shows errors for all riders simultaneously.
Recovery + Prevention
  1. 1
    Use jittered TTL: TTL = 30 + random(0, 10) seconds — staggers cache expiry
  2. 2
    Implement "probabilistic early expiry": recompute cache when TTL < 5s with 10% probability — prevents thundering herd
  3. 3
    Single-flight pattern: if cache miss, only one goroutine fetches from ClickHouse; others wait for the result

Data Quality — Cross-Cutting Failures
Scenarios that span multiple layers and are hardest to detect
CRITICAL
Silent data loss — pipeline appears healthy but events are dropping
Flink filter operator bug discards 5% of GPS events with no error, no alert, no DLQ entry
▼
What happens
A Flink filter operator meant to remove duplicate GPS pings has an off-by-one bug in its bloom filter time window. It discards 5% of valid events, not just duplicates. No exception is thrown. No DLQ entry. Kafka consumer lag is normal. Flink checkpoints succeed. From a pipeline health perspective, everything looks fine. But Silver GPS data has a 5% hole — ETA calculations are subtly wrong, driver distance calculations undercount, earnings are slightly off. This could go undetected for days or weeks.
Detection — this requires proactive checks
This failure type cannot be detected by infrastructure monitoring alone. It requires data-level checks:
-- Run hourly: compare event count between layers SELECT bronze_count, silver_count, (bronze_count - silver_count) / bronze_count AS drop_rate FROM ( SELECT count(*) AS bronze_count FROM bronze_gps WHERE hour = current_hour ) b, ( SELECT count(*) AS silver_count FROM silver_gps_validated WHERE hour = current_hour ) s -- Alert if drop_rate > 0.02 (2% expected for legitimate anomalies)
Recovery steps
  1. 1
    Identify the buggy Flink operator from the count discrepancy — binary search across operators
  2. 2
    Fix and redeploy Flink job
  3. 3
    Backfill: re-run Flink batch job over Bronze for the affected time range to regenerate correct Silver
  4. 4
    Recalculate Gold aggregates and ClickHouse partitions for the affected window
HIGH
Time zone confusion — all timestamps shifted by ±5.5 hours in Gold
Mixed UTC and IST timestamps in Gold Parquet → zone heatmaps show peak demand at wrong hours
▼
What happens
The GPS devices send timestamps in local time (IST = UTC+5:30) without timezone info. The CDC events from PostgreSQL use UTC. Flink processes both streams and joins them. A join condition uses trip_start from PostgreSQL (UTC) to match GPS pings (IST). All GPS pings for a trip are "5.5 hours early" — they appear to be from before the trip started. The join fails silently (no matching window). Gold heatmaps show zero GPS activity during actual peak hours.
Prevention
All timestamps stored as UTC in every system GPS device SDK: convert to UTC before transmit Schema: all timestamp fields annotated as UTC in Avro doc field Flink: validate event_time is within ±24h of server_ingest_time
